# Qwen3.6-35B-A3B vs Gemini Flash Lite — A/B for /pregunta synthesis

**Status:** smoke complete, full eval deferred to FP8 cluster
**Date:** 2026-05-01
**Author:** automated session, judged by Claude Opus 4.7
**Branch:** `fix/boe-rangos-and-clean-anchors` (WIP stashed during eval)

## Verdict

**Qwen3.6-35B-A3B with a tuned prompt beats Gemini Flash Lite on the smoke set, even running on Q4_K_M local quantisation.**

Quality scores (8 retrieval-ready Q × 4-dim rubric × 2 pts = 64 max):
- Gemini Flash Lite (control): **48 / 64 = 75%**
- Qwen-portado (Gemini prompt verbatim): **49 / 64 = 77%**
- Qwen-tuneado (Qwen-tailored prompt): **57 / 64 = 89%**

**Qwen-tuned is +19% over Gemini, +16% over the naive port.**

The 2 out-of-scope questions (weather, prompt injection) are declined upstream
by retrieval (`reason=non_legal`) before any LLM call — both backends produce
identical canned responses.

## Context

Production currently uses `google/gemini-2.5-flash-lite` for `/v1/ask`.
A prior 2026-04 A/B chose Gemini Flash Lite over `qwen/qwen-turbo`,
`qwen/qwen3-vl-32b-instruct`, and `qwen/qwen3-next-80b-a3b-instruct`. **None
of these is Qwen3.6-35B-A3B** — the architecture announced for the upcoming
self-hosted cluster (FP8, vLLM/SGLang on RTX PRO 6000 Blackwell, 256K
context, native reasoning). This run is the first head-to-head between
production Gemini and the Qwen3.6 generation.

## Hardware context — why this is a lower bound

- **Local (this run):** Qwen3.6-35B-A3B-UD-**Q4_K_M** on Mac via llama.cpp,
  ~70 tok/s output, ~150 tok/s prompt processing.
- **Production target:** RTX PRO 6000 Blackwell, **FP8**, vLLM/SGLang.

Three penalties this local run absorbs vs the future server:

1. **Quantisation:** Q4 (~4-bit) discards more weight precision than FP8
   (~8-bit). Symptom we observed: Qwen rumiates the system-prompt rules
   inside reasoning ("I will follow this order strictly: ARTÍCULO PRINCIPAL
   primero…") — a sign the model can't internalise instructions at one
   pass. FP8 should recall them without re-reading.
2. **Attention quality at long context:** llama.cpp at 256K config + Q4 KV
   shows t/s degrading linearly with KV size (we saw 60→40 t/s as reasoning
   grew). vLLM/SGLang use paged attention and hold throughput flat.
3. **Prompt-eval latency:** ~150 t/s local CPU/Metal → 7K-token evidence
   takes ~25s of prompt eval before the first token. Server GPU does this
   at >3000 t/s, making TTFT sub-second. A longer, more prescriptive
   prompt becomes effectively free — i.e. tuning the prompt for quality
   stops trading off latency.

**Implication:** if local Q4 ties or beats Gemini, the FP8 server is
strictly better in both quality and speed.

## Methodology

For each question:

1. Retrieval runs **once** via the API's eval endpoint (`POST /v1/_eval/retrieval`).
   The new endpoint (added in `pipeline.ts:evalRetrieval` + `routes/ask.ts`)
   returns articles + evidenceText + systemPrompt without calling synthesis,
   so all variants see byte-identical input.
2. Both backends stream synthesis in parallel:
   - **Gemini:** OpenRouter `google/gemini-2.5-flash-lite`, temp 0.2,
     max_tokens 4000, `SYSTEM_PROMPT_STREAM` from prod synthesis.ts.
   - **Qwen:** local llama.cpp at `127.0.0.1:8080`, OpenAI-compatible,
     same prompt and temperature, max_tokens 16000 (reasoning needs
     budget).
3. Outputs persisted incrementally (`qwen-ab-smoke-results.json`).
4. **Variant 2 (Qwen-tuneado):** `qwen-ab-tuned.ts` re-runs Qwen against the
   same cached evidence with a Qwen-tailored prompt
   (`qwen-ab-tuned-results.json`).
5. Judging: Claude Opus 4.7 reads each answer in this session and scores
   on 4 dimensions (factual / citations / tone / completeness, 0-2 each).

The smoke set: 5 easy (vacaciones, fianza, paternidad, despido, alquiler dur.),
1 cross-law (embarazo), 1 constitutional (móvil/policía), 1 temporal
(renta history), 2 out-of-scope.

## What the tuned prompt did differently

[`packages/api/research/qwen-ab-tuned.ts`](qwen-ab-tuned.ts) defines
`TUNED_SYSTEM_PROMPT`. Diffs vs `SYSTEM_PROMPT_STREAM`:

1. **Strong opening "answer contract"** at the top: first word is "Sí" / "No"
   / "Depende" — short and binding. Fixes the hedge-before-answer pattern
   we saw in Qwen-portado (Q3).
2. **Priorisation block** with explicit anti-leak rule: "if ARTÍCULO
   PRINCIPAL is sectoral/regional but a general state law is also in the
   evidence pool, lead with the state law and treat the principal as
   exception." Fixes Q1 (where retrieval served funcionariado articles
   first).
3. **Completeness checklist** demanding key cuantías/plazos before
   closing — including "5-year LAU minimum, 24-month despido cap." Fixes
   Q5 directly.
4. **Anti-rumination** instruction: "do not narrate the rules to yourself
   in reasoning, do not recite each article verbatim, identify the main
   article and write." Reduces reasoning by ~20-30% across Q1/Q3.
5. **Shorter, less repetitive** (1.5 KB vs prod 2.4 KB) — Qwen Q4 has
   fewer rules to internalise.

## Failure modes the tuned prompt introduced

The two regressions point at the same axis: **the tuned prompt is too
willing to under-cite and to hedge**.

- **Q6 (embarazo, cross-law):** Tuned cites 3 vs portado's 13. Loses
  detail on lactancia, riesgo, prestación LGSS — all valid citizen-relevant
  rights. The "completeness" rule fired on cuantías but didn't enforce
  breadth on cross-law topics.
- **Q7 (móvil/policía):** Tuned says "No tengo información en la
  legislación proporcionada" — technically defensible (CE 18 covers
  *comunicaciones*, not *physical device search*) but useless for the
  citizen who wants a direct "No, requiere orden judicial."

Likely fix: add explicit "for cross-law questions, list ALL right
categories present in the evidence; do not prune to one bullet" and
"never decline if the evidence partially answers — say what it covers."

## Caveats — what this smoke does NOT prove

1. **N=8 retrieval-ready questions** is not statistically robust. Need
   the full 65-Q gold set (`spike-questions.ts` + `spike-questions-hard.ts`)
   to get tight error bars.
2. **Q4 quantisation noise:** quality numbers will improve on FP8.
   We can't yet predict whether FP8 closes Q6/Q7 regressions or just
   makes everything faster.
3. **Same retrieval bug for both:** Q1's retrieval pulls funcionariado
   articles ahead of the ET. Tuned worked around it via prompt;
   long-term fix is in retrieval (issue #40 — hybrid BM25+semantic).
4. **Same eval evidence:** byte-identical evidence isolates synthesis
   quality. But it doesn't measure the *combined* effect of tokens-
   unlimited Qwen pipelines (multi-pass synthesis, query rewriting,
   reranker FT) that the cluster will enable.

## Decision

**Do not switch production to Qwen yet.** Defer to the FP8 cluster:

1. When the cluster is live, re-run **the full 65-Q eval** with
   `qwen-ab-smoke.ts` pointed at the FP8 endpoint. Same gold set,
   same harness, same judge.
2. Iterate the tuned prompt to address Q6/Q7 regressions (cross-law
   breadth, no-hedge-on-partial-evidence).
3. If FP8 + tuned prompt holds the +19% quality lead at N=65, migrate
   `SYNTHESIS_MODEL` in `synthesis.ts` to the cluster endpoint behind
   a feature flag, A/B in production with shadow traffic, then cutover.

## Followups

- Issue #40 (hybrid BM25+semantic retrieval) — fixes the Q1 retrieval
  bug at the source instead of in the synthesis prompt.
- Issue #44 (reranker fine-tune) — frees up 1000-req/month Cohere
  budget once self-hosted; orthogonal to this A/B but unblocks volume.
- Token economy: Qwen burns ~3500-5000 tokens of reasoning per query.
  At cluster pricing (free), this is fine. If we ever need to bound
  cost, add `/no_think` mode toggle for non-citizen-facing internal
  uses.
