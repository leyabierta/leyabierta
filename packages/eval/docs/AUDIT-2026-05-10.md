# Auditoría de datasets de evaluación existentes

**Fecha:** 2026-05-10
**Autor:** Devin (orquestador) + Alex
**Contexto:** Antes de generar un dataset nuevo de ~2000 preguntas con un pipeline multi-agente sobre NaN, auditamos lo que ya existe en el repo para decidir qué reciclar.

## TL;DR

- En el repo hay **dos** conjuntos de preguntas de evaluación reales, no 24+ como parecía.
- Los 24 ficheros `data/eval-answers-*.json` son **24 *runs* de RAG sobre el mismo ~65-question set**, no 24 datasets distintos.
- Conjunto canónico de retrieval (citizen voice): `packages/api/research/datasets/citizen-queries.json` — **50 preguntas**.
- Conjunto canónico de RAG (registro formal): embedido en `data/eval-answers-*.json` — **64 preguntas únicas**.
- **Total de seeds humanos disponibles: ~114 preguntas únicas**, claramente insuficiente para el objetivo de 2000.
- **Veredicto: el pipeline de generación está justificado.** Las 114 sirven como anclajes de calibración y como semillas para diversificar, no como base mayoritaria.

## Inventario detallado

### A. citizen-queries.json (retrieval eval, registro ciudadano)

| Campo | Valor |
|---|---|
| Path | `packages/api/research/datasets/citizen-queries.json` |
| Versión | v2 (2026-04-27) |
| Tamaño | 50 preguntas |
| Schema | `id, question, category, expectedNorms[], rationale` |
| Granularidad de ground truth | Norma (no artículo) |
| Multi-respuesta | Sí (varias normas por pregunta cuando hay alternativas legítimas) |
| Voz | Ciudadana, minúsculas, sin tildes, sin puntuación, jerga cero |
| Categorías | 20 materias, fuerte sesgo a `trabajo` (13/50 = 26%) |
| Métricas | Recall@1 / @5 / @10 a nivel norma |
| Uso actual | Eval principal de Qwen vs Gemini, A/B Phase 4, ablaciones de hybrid search |

**Distribución por categoría:**

| Categoría | n |
|---|---|
| trabajo | 13 |
| familia, seguridad-social, vivienda-arrendamiento | 4 c/u |
| salud | 3 |
| administrativo, consumo, datos-personales, extranjeria, penal, violencia-genero, vivienda-comunidad | 2 c/u |
| 8 categorías más | 1 c/u |

**Calidad heurística:**

- Cero leaks de BOE-ID en el texto de la pregunta.
- Voz ciudadana auténtica (verificado a mano sobre muestreo de 10).
- v2 expandió `expectedNorms` para admitir respuestas alternativas legítimas detectadas durante ablations — patrón sólido que conviene mantener.

**Limitaciones:**

- n=50 — un swing de 2 puntos de R@5 está dentro del ruido.
- Sin `expectedArticles[]` (granularidad solo norma). Los embeddings son por artículo; la métrica natural debería ser también por artículo.
- Desbalance fuerte hacia laboral.
- Categorías de materia son ad-hoc, no las materias oficiales BOE.
- Tuning (`tune-eval.ts`, `fast-tune.ts`) y eval final usan el mismo set — riesgo de overfit.

### B. eval-answers-*.json (RAG eval runs, registro formal)

| Métrica | Valor |
|---|---|
| Ficheros | 24 |
| Filas totales | 1,381 |
| **Preguntas únicas** | **64** |
| Schema por fila | `id, question, category, expectedAnswer, expectedNorms[], ragAnswer, ragCitations[], ragDeclined, ragLatencyMs, ragModel` |
| Granularidad ground truth | Norma (`expectedNorms[]`) + texto libre (`expectedAnswer`) |
| Voz | Formal: "¿Cuántos días de vacaciones me corresponden al año?", "¿Me puede subir el alquiler mi casero cuando quiera?" |
| Categorías | 3 funcionales: `clear` (38), `cross-law` (21), `out-of-scope` (5) — no son materias |

**Por qué hay 24 ficheros con casi las mismas preguntas:**

Cada fichero es la salida de una variante de RAG distinta sobre el mismo set:

- `top500*` — variantes de retrieval con top-500 candidates
- `p1-staleness` / `p1-v2-robust` — Phase 1 fixes
- `p2-conditional` / `p2-contextual` / `p2-query-expansion` — Phase 2 ablations
- `p3-statelaw` — Phase 3
- `p4-prompt-rewrite` — Phase 4
- `504-omnibus` / `504-laws` / `504-diversity` — variantes de pool size
- `484k-v2`, `484k` — full corpus runs

Cada fichero conserva `ragAnswer`, `ragCitations`, `ragLatencyMs`, `ragModel`. Son **outputs**, no inputs.

**Distribución de runs por pregunta:** 36 preguntas aparecen en las 24 runs; el resto en menos (algunas se añadieron tarde, otras se quitaron).

**Calidad heurística:**

- Cero leaks de BOE-ID en el texto.
- 18/64 preguntas (28%) tienen `expectedNorms` con >1 norma, patrón compatible con el del citizen set.
- `expectedAnswer` es texto libre verificado a mano — útil como ancla para sintesis eval, no para retrieval.
- Las citas de RAG (`ragCitations`) llevan `articleTitle` y `verified` boolean. Eso sugiere que internamente se valida contra el corpus, pero **el ground truth almacenado sigue siendo a nivel norma**.

## Solapamiento entre los dos sets

Las 50 (citizen) y las 64 (RAG) cubren temas similares (vivienda, trabajo, familia) pero con voz radicalmente distinta. Son **complementarias, no redundantes**:

- Citizen (50): `"el casero quiere echarme de casa"`
- RAG (64): `"¿Cuál es el plazo legal para que un casero pueda iniciar un proceso de desahucio?"`

Si fusionamos sin más, el promedio del registro queda artificialmente alto. Lo correcto es etiquetar `voice: "citizen" | "formal"` y poder evaluar ambos por separado.

## Veredicto por origen

| Origen | Acción | Razón |
|---|---|---|
| `citizen-queries.json` (50) | **Reciclar como gold de calibración** | Voz ciudadana auténtica, ground truth verificado, ya alineado con métricas de retrieval. |
| `eval-answers-*.json` (64 únicas) | **Extraer las 64 únicas, etiquetar `voice: formal`, reciclar como semillas** | Buena voz formal, expectedNorms decente, expectedAnswer útil para futuras evals de síntesis. |
| Las otras 23 *runs* | **Archivar (no borrar)** | Son experimentos históricos. Quedan en `data/` por trazabilidad pero no son fuente de preguntas. |

Total reutilizable: **114 preguntas humanas**. El generador agregará ~1900 más para llegar a 2000.

## Recomendaciones para el dataset v3

1. **Schema unificado** con campos nuevos:
   - `voice: "citizen" | "formal"`
   - `expectedArticles: { norm: string; article: string }[]` (nuevo: granularidad de artículo)
   - `provenance: { source: "human-citizen" | "human-rag" | "agent-generated"; ... }`
   - `split: "train" | "val" | "test"` con disjunción por norma.
2. **Mantener el patrón multi-respuesta de v2** (varias `expectedNorms` legítimas).
3. **Estratificar por materia oficial BOE** (no las categorías ad-hoc de citizen-queries).
4. **Anotar las 114 humanas a nivel artículo** (puede hacer el pipeline mismo, vía retrieval + juez, antes de generar las nuevas — buena prueba de fuego del pipeline).
5. **Held-out de 50 preguntas humanas** (25 citizen + 25 RAG) que el generador no ve, como test final externo al loop.
